# 技術スタック

## アーキテクチャ

ハイブリッドデータ指向アプローチ:
- **コアシミュレーション**: データ指向のティックシステムで動作（資源フロー、ベルト輸送、機械加工）
- **レンダリング＆入力**: Godotのシーンツリーで処理（Camera、Input、シミュレーション-グラフィックスブリッジ）
- **UI**: GodotのControlノード経由（デバイスパレット、納品カウンター、HUD）

分離原則: ゲームロジックはSceneTree/Node APIに依存しない純粋なデータ＋システム。Godotノードはレンダリングと入力用の薄いアダプター。

シミュレーション制約:
- **固定ティック**: `_physics_process`（デフォルト60Hz）でシミュレーション更新。可変フレームレートの`_process`はプレゼンテーション専用
- **決定性**: 同一入力で同一結果を保証。乱数使用時はシード管理必須
- **セーブ/ロード**: ゲーム状態はResourceベースでシリアライズ。シーンツリー状態に依存しない

## コア技術

- **エンジン**: Godot 4.3+
- **言語**: GDScript
- **プラットフォーム**: Linux/WSL2ネイティブ開発、クロスプラットフォームエクスポート

## 開発標準

### 静的解析
- **gdlint**: gdtoolkitベースのスタイルチェッカー。設定は`godot/.gdlintrc`（max-line-length: 120, max-public-methods: 40, addons除外）
- **型注釈チェッカー**: `scripts/check_gdscript_types.py`。関数引数・戻り値・クラス変数の型注釈を強制（ローカル変数は警告のみ）
- **統一ランナー**: `scripts/gdcheck.sh`（gdlint + 型チェック。`--all`でCI用、ファイル指定で単体チェック）
- **Claude Code hook**: `.gd`ファイルのWrite/Edit時にPostToolUseで自動実行。エラーはstderr経由でClaude にフィードバック
- **Stop hook (テストゲート)**: Claude停止時に`.gd`変更を検出し、変更がある場合は全テスト実行。失敗時はexit 2でClaudeに修正を強制。`SKIP_TEST_GATE=1`でバイパス可能。`.claude/hooks/test-gate.sh`

### コード品質
- GDScript静的型付けを使用（`var x: int`, `func foo() -> void`）。`Variant`型の使用は意図的な場合のみ
- ゲームロジックをSceneTree/Node APIから分離（テストはGdUnit4経由で`xvfb-run`+X11仮想ディスプレイ上で実行）
- ゲーム状態にグローバルシングルトンを使わない。依存性注入を使用。autoloadはイベントバス・設定管理等の横断的関心事のみ許可
- シミュレーション内はティック順序で同期実行。シグナル/イベントバスはプレゼンテーション層↔ロジック層の通知に使用

### テスト
- Layer 1（ユニット）: SceneTree非依存の純粋ロジッククラスをCLIテスト（xvfb-run経由）
- Layer 2（インテグレーション）: Godot内でのコンポーネント間テスト
- Layer 3（E2Eテスト）: スクショ+AI視覚評価によるフルシーン自動検証（SceneRunner使用）
- Layer 4（ヒューマンレビュー）: 主観的品質・ゲームフィールを手動検証

## 主要な技術的意思決定

- **GodotをUnityより優先**: ネイティブLinux/WSL2サポートにより、複雑なbind-mountやMCPブリッジの回避策が不要
- **ハイブリッドデータ指向**: シミュレーションティックには純粋なデータ＋システム（GDScript）。Godotシーンツリーはプレゼンテーション専用
- **タイルベースワールド**: 工場レイアウトにグリッド座標を使用。レンダリングにはTileMapLayer

### タイルマップ実装方針

3層構成:

1. **TileSet（静的定義）**: 地形種別・資源種別等の`custom_data_layers`を持つ静的タイル定義。`Resource`として`@export`プロパティ+安定IDで定義
2. **Core Grid（RefCounted）**: 密グリッドには`PackedInt32Array`等のPacked配列、疎な動的状態（占有、機械ID）には`Dictionary<Vector2i, GridCellData>`。SceneTree非依存
3. **TileMapLayer（レンダリング専用）**: Core Gridからの一方向同期のみ

※`GridCellData`は独自クラス名（Godot組み込みの`TileData`との名前衝突を回避）

### 主要設計判断

- **データフロー（単方向）**: 入力(Node) → コマンド → シミュレーション(RefCounted) → 共有データ → 描画/UI(Node)
- **シミュレーション駆動ルール**: `_physics_process`でコア状態を変更するNodeは1つのみ。プレゼンテーション層は観察・ミラーのみ
- **シグナル使用範囲**: プレゼンテーション↔ロジック間の通知のみ。シミュレーション内部ではシグナルを使わずティック順序で同期実行
- **ベルトシステム**: スロットベース(4スロット/タイル)、ItemType+Progress格納、チェーン末尾→先頭処理順、バックプレッシャー（次が満杯なら停止）。ベルト接続グラフは配置/撤去時のみダーティ再構築（毎ティック再構築しない）
- **機械ポートI/O**: 相対オフセットで入出力ポート定義、隣接ベルトの方向で空間的に接続解決、明示的リンク不要
- **回転**: 4方向(N/E/S/W)、int enum、ポートは北基準で定義しクエリ時に回転
- **資源モデル**: 無限採取（枯渇しない）
- **パフォーマンス方針**: タイルは個別ノード化しない、ベルト上アイテムも個別ノード化しない（スロットバッファ管理）。ベルトアイテム描画は`MultiMeshInstance2D`またはスプライトプールを推奨
- **セーブ/ロード境界**: RefCountedランタイム状態を`Resource`スナップショットに変換してシリアライズ。シーンツリー状態には依存しない
